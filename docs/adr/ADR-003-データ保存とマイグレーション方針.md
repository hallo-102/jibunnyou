# ADR-003: データ保存とマイグレーション方針

- 状態: 採用
- 決定日: 2026-07-10

## 背景

現行BackendはSQLAlchemy `create_all`でテーブルを作成するが、設計はPostgreSQL 16、版管理されたMigration、run単位の不変履歴を要求する。現行モデルには既に部分データが存在し得るため、全面再作成は危険である。

## 検討した選択肢

1. `create_all`を継続する。
2. DBを削除して詳細設計どおり一括再作成する。
3. Alembicを導入し、現行モデルをベースラインとして段階的に拡張する。

## 採用した方法

Alembicを導入し、現行スキーマを初期ベースラインとして記録した後、Artifact、監査、AI独立分析、状態履歴を安全な追加Migrationで実装する。破壊的変更はExpand/Contract方式を使う。

## 採用理由

- 既存データを保護できる。
- DB仕様と実装の差を追跡できる。
- ロールバック・ステージング試験が可能になる。

## 利点

- リリースとDB版を紐付けられる。
- 将来のNAS更新が再現可能になる。
- 一括置換によるデータ損失を避けられる。

## 欠点

- 現行スキーマと詳細設計のマッピング作業が必要である。
- downgrade不能なデータ変換では別途バックアップが必要である。

## 影響範囲

- `backend/alembic/`
- `backend/alembic.ini`
- `backend/app/db/models.py`
- Docker起動手順、バックアップ、テスト

## 移行・後方互換性

既存テーブルを即時削除・改名しない。新カラム・新テーブルを先に追加し、読み書きを切り替え、検証後に不要部分を将来のMigrationで整理する。
