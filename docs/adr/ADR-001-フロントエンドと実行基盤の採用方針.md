# ADR-001: フロントエンドと実行基盤の採用方針

- 状態: 採用
- 決定日: 2026-07-10

## 背景

設計書にはReact + Vite + PWAとNext.js + TypeScriptの両方が記載されている。現行の未コミット実装はNext.js 15、FastAPI、PostgreSQL、Redis、Celery、Nginxで構成され、Frontendのproduction buildは成功している。

## 検討した選択肢

1. 現行Next.jsを破棄してViteで再作成する。
2. 現行Next.jsを継続し、最新UI仕様へ段階的に分割・改善する。
3. 両方を並行維持する。

## 採用した方法

Next.js 15 + TypeScriptをFrontendの正式構成として継続する。BackendはFastAPI、正式DBはPostgreSQL、ジョブはRedis + Celery、入口はNginx、NASではDocker Composeを採用する。

## 採用理由

- 既存資産を最大限利用する原則に合う。
- Frontend buildが実際に成功している。
- API契約を維持したままページ・コンポーネント分割ができる。
- Viteへの全面移行は機能価値を増やさず回帰リスクだけを増やす。

## 利点

- 現行コードを捨てずに進められる。
- App Router、server/client分離、standalone buildを利用できる。
- 段階移行が可能である。

## 欠点

- Vite前提の古い設計例はそのまま利用できない。
- 現行の単一巨大ページは大幅なコンポーネント分割が必要である。

## 影響範囲

- `frontend/`
- Frontend Dockerfile
- Nginx設定
- UI設計内のVite/PWA記述

## 移行・後方互換性

既存の `/` はダッシュボードとして維持する。APIのURL・JSON契約はFrontend技術と切り離す。PWAはVer.1.0の必須範囲を満たした後の追加候補とする。
