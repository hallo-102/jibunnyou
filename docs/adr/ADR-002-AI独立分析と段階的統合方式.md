# ADR-002: AI独立分析と段階的統合方式

- 状態: 採用
- 決定日: 2026-07-10

## 背景

2026-07-08以前の設計と現行コードは、AIがPython順位、score、期待値、危険馬フラグを見て上げ下げするセカンドオピニオン方式である。この方式ではAIがPython結果を言い換えるだけになりやすい。2026-07-10改訂版とマスタープロンプトは、Python最終順位を見ない独立分析を必須としている。

## 検討した選択肢

1. 現行のPython可視セカンドオピニオンを継続する。
2. AIへ全データを渡し、自由に統合順位を直接作らせる。
3. 独立分析を先に不変保存し、その後でPython比較・統合を別処理にする。

## 採用した方法

次の4段階を正式方式とする。

1. Python最終順位・score・印・買い目・買い判断を含めないAI独立分析。
2. 独立結果をハッシュ付きで固定保存。
3. Python結果を開示して一致、不一致、理由差、反対材料を比較。
4. 統合順位、根拠、信頼度、不確実性を作成。

## 採用理由

- AIの独立価値を検証できる。
- Pythonへのアンカリングを減らせる。
- 独立結果と統合判断を別々にバックテストできる。
- 最新仕様の優先規則に合う。

## 利点

- Python単独、AI単独、統合を同条件で比較できる。
- AIがどの時点で何を見たか監査できる。
- 差分の捏造を避けられる。

## 欠点

- AI run、Schema、プロンプト、画面、テストが増える。
- API利用コストと実行時間が増える可能性がある。
- Pythonで計算した基礎特徴量のうち、何を独立入力に許可するかを明示する必要がある。

## 影響範囲

- `backend/app/services/ai_opinion.py`
- `backend/app/services/ai_independent.py`
- `backend/app/services/ai_integration.py`
- AI API・Schema・DB migration
- AI分析・比較・統合UI
- 品質チェック、テスト、バックテスト

## 移行・後方互換性

既存の `ai_second_opinion` runは削除しない。旧方式として閲覧可能にし、`analysis_stage`、`python_result_visible`、`input_hash`を持つ新runへ移行する。旧結果を新しい独立結果として再分類しない。
