# Keiba AI Studio 運用手順

- 対象版: 1.1.0
- 対象環境: Windows 11 + Docker Desktop + Docker Compose v2
- 公開範囲: ローカルPCのGateway `127.0.0.1:18080`を想定

## 1. インストール

1. Docker Desktopをインストールし、Linux container modeで起動する。
2. 本プロジェクトを日本語・OneDriveを含む現在の場所へ配置する。
3. PowerShellでプロジェクト直下へ移動する。
4. `Copy-Item .env.example .env`を実行する。
5. `.env`の`KEIBA_POSTGRES_PASSWORD`を長いランダム値へ変更し、`KEIBA_DATABASE_URL`内も同じ値へ変更する。
6. AIを使用する場合だけ`OPENAI_API_KEY`を設定する。キーを画面、Git、ログ、文書へ貼らない。

## 2. 初回起動

```powershell
# Compose解決結果を検査する。秘密値を共有画面へ出力しない。
docker compose config --quiet

# Database、queue、API、Worker、Beat、Frontend、Gatewayを起動する。
docker compose up -d --build

# 全serviceの状態を確認する。
docker compose ps

# 依存を含むreadinessを確認する。
Invoke-RestMethod http://127.0.0.1:18080/api/ready
```

`checks`の`postgres`、`redis`、`artifact_storage`、`config`がすべて`ok`なら、`http://127.0.0.1:18080`を開く。

## 3. 通常操作

1. 開催日を選択する。
2. レース・オッズの取得状態と品質を確認する。
3. Python予想を実行する。
4. Python情報を含まない独立AIを実行し、固定済み表示を確認する。
5. Python/AI比較・統合を実行する。
6. 予算・券種・方式を確認し、買い目候補だけを保存する。
7. 外部で購入した場合だけ、明示確認後に手動購入記録を行う。
8. 確定結果を取得し、精算、成績分析を確認する。

品質RED、重大不一致、上限超過、暫定結果では安全側へ停止する。自動投票は常に無効である。

## 4. 停止・再開

```powershell
# 通常停止。DB/Redis volumeは保持する。
docker compose stop

# 再開する。
docker compose start

# containerを作り直せる状態で停止する。volumeは保持する。
docker compose down
```

停止前にrunning jobがないことを画面の「ジョブ・品質」で確認する。`down -v`は運用手順に含めない。

## 5. 更新

```powershell
# 変更前backupを取得する。
.\infra\operations\Backup-KeibaAiStudio.ps1

# 新しいsource取得後にimageを再buildする。
docker compose build --pull

# Migrationを適用する。
docker compose run --rm api alembic upgrade head

# 全serviceを更新する。
docker compose up -d

# 更新後検証を行う。
Invoke-RestMethod http://127.0.0.1:18080/api/ready
$api = docker compose ps -q api
docker exec $api python -m pytest tests -q
```

Migration revisionが想定と違う場合は`stamp`で合わせず、更新を中止してバックアップから隔離復元する。

## 6. バックアップ

通常バックアップはDB、Raw、master、exports、logs、Manifest、SHA-256を同じ`backup_id`へ保存する。

```powershell
# .backups配下へ完全backupを取得する。
$backup = .\infra\operations\Backup-KeibaAiStudio.ps1
$backup

# DBだけの短時間検証backupを取得する。
.\infra\operations\Backup-KeibaAiStudio.ps1 -DatabaseOnly

# 別ドライブへ保存する。
.\infra\operations\Backup-KeibaAiStudio.ps1 -Destination 'E:\KeibaAiStudioBackups'
```

実行前にrunning jobがないことを確認する。日次30世代、月次24世代を目安に別媒体へ保管する。`.env`とAPIキーはバックアップへ同梱しない。

## 7. 復元試験

最初は必ず隔離DBへ復元し、稼働DBを変更しない。

```powershell
# Manifestと全hashを検証し、隔離DBへ復元する。
.\infra\operations\Restore-KeibaAiStudio.ps1 `
  -BackupPath '.backups\backup_YYYYMMDD_HHMMSS_JST' `
  -TargetDatabase 'keiba_restore_test' `
  -ConfirmRestore
```

稼働DBへ戻す場合はAPI、Worker、Beatを停止し、復元先、バックアップID、影響範囲を再確認してから次を実行する。

```powershell
# 重要: 稼働DBを書き換える明示指定。事前backupが必要。
docker compose stop api worker beat frontend gateway
.\infra\operations\Restore-KeibaAiStudio.ps1 `
  -BackupPath 'E:\KeibaAiStudioBackups\backup_YYYYMMDD_HHMMSS_JST' `
  -TargetDatabase 'keiba_ai_studio' `
  -ConfirmRestore `
  -AllowPrimaryDatabase `
  -RestoreFiles
docker compose start api worker beat frontend gateway
Invoke-RestMethod http://127.0.0.1:18080/api/ready
```

復元後はrevision、代表run、買い目履歴、監査ログ、Artifact hashを確認する。

## 8. ログ確認

```powershell
# 全serviceの直近logを確認する。
docker compose logs --since 30m

# APIとWorkerを追跡する。
docker compose logs -f --tail 200 api worker

# request IDで構造化logを検索する。
docker compose logs api | Select-String '<X-Request-ID>'
```

画面の失敗理由とrequest ID、対象日、race ID、job IDを記録する。ログを共有する前にAPIキー、password、session情報がないことを確認する。

## 9. 設定・モデル変更

- `.env`: service共通の接続、timeout、retry、AI modelを変更する。
- 買い目画面: source、券種、方式、1点金額、レース/日上限、最大点数を変更する。
- Python model: 既存予想資産と版数を変更後、Golden Masterと3か月backtestを必ず実行する。
- AI prompt/schema: 版数を上げ、独立入力禁止key、構造化出力、比較統合上限の全テストを実行する。

`.env`変更後は対象serviceを`docker compose up -d --force-recreate ...`で再作成する。自動投票の有効化は設定項目として提供しない。

## 10. API・DB確認

- Swagger UI: `http://127.0.0.1:18080/api/docs`
- OpenAPI: `http://127.0.0.1:18080/api/openapi.json`
- Health: `/api/health`
- Readiness: `/api/ready`
- Version: `/api/version`
- Alembic: `docker compose run --rm api alembic current`

API、DB、画面の契約は統合仕様書と要件トレーサビリティを正とする。
